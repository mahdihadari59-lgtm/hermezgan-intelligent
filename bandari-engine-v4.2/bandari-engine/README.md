# Bandari Engine v4.0

موتور پردازش و ترجمه گویش بندری — بازسازی‌شده با معماری ۸ لایه‌ای کامل (برخلاف v3.0 که فقط ۴ لایه از ۸ لایه‌ی طراحی‌شده را داشت).

## چه چیزی نسبت به v3.0 اضافه/اصلاح شد

**لایه‌های جدید (قبلاً در دیاگرام بودند ولی پیاده نشده بودند):**
- `dialect/` — تشخیص گویش (بندری/میناب/قشمی/بستکی) بر اساس نشانگرهای واژگانی
- `morphology/` — تشخیص سادهٔ ریشهٔ فعل و زمان دستوری
- `context/` — حافظهٔ مکالمه per-session با TTL و پاک‌سازی خودکار
- `llm/` — آداپتور مترجم LLM، pluggable و به‌صورت پیش‌فرض غیرفعال (بدون نیاز به کلید API برای اجرا)

**اصلاح باگ‌ها:**
- در v3.0، اندپوینت `/api/learn` مستقیماً `dict.words.set` را صدا می‌زد و انکپسولیشن کلاس را می‌شکست؛ و واژه‌های یادگرفته‌شده با ری‌استارت سرور از بین می‌رفتند. حالا `Dictionary.addWord()` واژه را هم در حافظه و هم در `data/learned_words.json` ذخیره می‌کند.
- تطبیق تقریبی (fuzzy match) دیکشنری قبلاً برای کلمات خیلی کوتاه (۱-۲ حرفی) هم اجرا می‌شد و می‌توانست تطابق‌های کاذب بدهد؛ محدود به کلمات ≥۳ حرف شد.
- `index.js` قبلی همهٔ روت‌ها را مستقیم داخل فایل اصلی تعریف کرده بود؛ حالا در `api/routes.js` جدا شده و با dependency injection موتورها را می‌گیرد (تست‌پذیرتر).
- یک `utils/logger.js` واقعی اضافه شد به‌جای `console.log` پراکنده.

## ساختار

```
bandari-engine/
├── index.js              # نقطه ورود، سیم‌کشی ۸ لایه

├── dictionary/             # لایه ۲: دیکشنری + یادگیری پایدار
├── dialect/                 # لایه ۱: تشخیص گویش
├── morphology/           # لایه ۳: ریخت‌شناسی
├── grammar/               # لایه ۴: قواعد دستوری
├── intent/                  # لایه ۵: تشخیص نیت
├── context/               # لایه ۶: حافظه مکالمه
├── rag/                      # لایه ۷: دانش‌نامه
├── llm/                      # لایه ۸: مترجم هوشمند (اختیاری)

```

## اجرا (روی Termux)

```bash
cd bandari-engine
npm install
npm start
# سرور روی http://localhost:5200 بالا می‌آید
```

اگر شبکه/اینترنت برای npm install در دسترس نیست، همان زمانی که به شبکه وصل شدی نصب کن؛ کد بدون نیاز به تغییر آماده است.

## فعال‌سازی لایه LLM (اختیاری)

```bash
BANDARI_LLM_ENABLED=1 BANDARI_LLM_PROVIDER=anthropic npm start
```

بدنهٔ واقعی تماس با provider باید در `llm/index.js` داخل متد `_callProvider` نوشته شود؛ فعلاً عمداً خالی است تا سرویس بدون کلید API هم بالا بیاید.

## دیکشنری (v4.1) — گسترش دسته‌بندی‌شده

دیکشنری از ~۱۳۰ به ۲۴۴ واژه در ۱۵ دسته + ۸ عبارت + ۵ اصطلاح/ضرب‌المثل گسترش یافت:

`greetings, family, numbers, colors, body, food, sea, weather, time, verbs, adjectives, places, traffic, emotions, questions`

⚠️ **مهم:** این واژگان از دانش زبان‌شناختی دربارهٔ ویژگی‌های شناخته‌شدهٔ گویش بندری/هرمزگانی ساخته شده‌اند (نه از یک کورپوس یا گویشور بومی تأییدشده). قبل از استفادهٔ عمومی/انتشار، پیشنهاد می‌شود یک گویشور بومی (یا چند نفر از شهرهای مختلف چون تلفظ‌ها بین بندرعباس/میناب/قشم فرق دارد) این فایل را مرور کند: `dictionary/words.json`.

منبع تولید در `scripts/build_dictionary.js` نگه داشته شده — اگر واژه‌ای اشتباه بود یا خواستی دسته جدید اضافه کنی، همون‌جا ویرایش کن و دوباره اجرا کن:
```bash
node scripts/build_dictionary.js
```

### اندپوینت‌های جدید

```bash
curl http://localhost:5200/api/dictionary/categories
curl http://localhost:5200/api/dictionary/category/family
```

اصطلاحات/ضرب‌المثل‌ها به‌صورت خودکار در راه‌اندازی به لایه RAG اضافه می‌شوند (یعنی `/api/translate` می‌تونه بهشون جواب بده، نه فقط واژه‌به‌واژه ترجمه کنه).

## دیکشنری ۹-گویشی (v4.2) — جدید

علاوه بر دیکشنری بندری موجود (۲۴۴ واژه، `dictionary/`)، یک لایه کاملاً جدید اضافه شد:
`dictionary/multilingual.js` + `dictionary/multilingual_words.json`، که بین ۹ گویش/زبان
استان هرمزگان ترجمه دوطرفه انجام می‌ده: بندری (ban)، مینابی (min)، قشمی (qes)، جاسکی (jas)،
لنگه‌ای (lan)، بستکی (bas)، خمیری (kha)، رودانی (rud)، سیریکی (sir).

**۹۹ ورودی، در دو سطح کیفیت مجزا:**
- **۱۴ ورودی `verified`** — از مستندی که فرستادید با داده کامل ۹-گویشی + IPA + مثال + امتیاز اطمینان واقعی (۹۰-۹۵٪). این‌ها معتبرترین بخش دیکشنری‌اند.
- **۸۵ ورودی `sourced`** — از مستندات شهرستان‌ها/جزایر/فرهنگ/رانندگی که فرستادید، مستقیم رونویسی شدن (نه حدس من)، ولی فقط گویش بندری دارن (بقیه ۸ گویش داده نداشتن).

هر entry فیلد `data_quality` داره تا مشخص باشه کدوم ورودی چقدر قابل‌اتکاست.

### اندپوینت‌های جدید

```bash
# ترجمه یک واژه بین هر دو گویش/فارسی (source/target از: persian, ban, min, qes, jas, lan, bas, kha, rud, sir)
curl -X POST http://localhost:5200/api/multilingual/translate \
  -H "Content-Type: application/json" \
  -d '{"word":"آب","source":"persian","target":"ban"}'

# ترجمه به تمام گویش‌های موجود همزمان
curl -X POST http://localhost:5200/api/multilingual/translate-all \
  -H "Content-Type: application/json" \
  -d '{"word":"خرما"}'

# جستجوی یک عبارت در تمام گویش‌ها همزمان
curl "http://localhost:5200/api/multilingual/search?q=جاده"

# واژه‌های یک دسته در یک گویش خاص
curl "http://localhost:5200/api/multilingual/category/driving?dialect=ban"

# لیست همه گویش‌ها با تعداد واژه هرکدام
curl http://localhost:5200/api/multilingual/dialects
curl http://localhost:5200/api/multilingual/dialects/rud
```

### تشخیص گویش (dialect detector) هم به ۹ گویش گسترش یافت

نشانگرها از ویژگی شاخص هر گویش طبق جدول مستندات (مثلاً «تبدیل آب به هو» برای بندری، «تبدیل من به مو» برای رودانی) به‌علاوه واژه‌های واقعی دیکشنری. دو گویش (خمیری، سیریکی) در مستندات فقط توضیح کیفی داشتن («ترکیب با گویش‌های غربی»، «اصطلاحات شرقی») بدون نشانگر مشخص، پس فعلاً نشانگر ندارن — تشخیصشون ضعیف‌تر از بقیه‌ست تا داده واقعی‌تری اضافه بشه.

### محدودیت مهم که باید بدونید

از ۹۹ ورودی، فقط ۱۴ تا واقعاً ۹-گویشی‌ان. برای بقیه ۸۵ تا (`sourced`)، اگه بخواید بین دو گویش غیر از بندری ترجمه کنید (مثلاً مینابی↔قشمی) روی این ۸۵ واژه، پاسخ `no_data_for_target_dialect` می‌گیرید — چون داده‌ای براشون نیست، نه اینکه باگ باشه. برای گسترش واقعی به ۹ گویش کامل، باید هرکدوم از این ۸۵ واژه رو با یک گویشور از هر منطقه تکمیل کرد.

## نکات باز برای گام بعدی

- برای رسیدن به مقیاس ۵۰هزار+ که دیاگرام اصلی وعده داده، منبع geo.db (اسم شهرها/جزایر/محلات) مسیر جدایی از این دیکشنری ترجمه است — چون آن داده POI/مکان است نه جفت ترجمه. برای اضافه‌کردنش به‌عنوان دانش مکانی در RAG (نه دیکشنری)، لازمه فایل واقعی geodata رو آپلود کنی.
- `context/` فعلاً in-memory است؛ اگر خواستی بین ری‌استارت‌ها هم بماند، می‌شه مثل بقیه سرویس‌های HDP با یک SQLiteAdapter جایگزین کرد.
- تشخیص گویش صرفاً واژگانی است (نه آوایی)؛ برای دقت بیشتر باید نشانگرهای هر گویش را از دادهٔ واقعی گسترش داد.
- توصیه می‌شود قبل از استفادهٔ production، `dictionary/words.json` توسط گویشور بومی راستی‌آزمایی شود (به‌خصوص بخش خانواده و افعال که بیشترین ریسک خطای من در آن‌هاست).
